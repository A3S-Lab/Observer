# Linux 4.19 ARM64 Observer Hotfix Design

## Goal

Produce a narrowly scoped replacement for
`/opt/anysentry/observer/bin/a3s-observer-collector` that can load and attach its
legacy kprobes on the target UOS 20 ARM64 host:

- Linux `4.19.0-arm64-server`
- AArch64
- glibc 2.28
- 64 KiB pages

The hotfix must leave ClickHouse, the AnySentry API, configuration, and stored
data untouched. The first delivery is a standalone collector binary for manual
replacement and target-host validation. A complete `compat2` suite archive is
out of scope until this binary passes on the real 4.19 host.

## Confirmed Failure

The target host starts ClickHouse and the API successfully. The collector fails
when every legacy eBPF program is loaded and exits with:

```text
Object BTF couldn't be loaded in the kernel
[2] ENUM c_void size=1 vlen=2 Expected size:4
legacy probe load failed ... BPF_PROG_LOAD syscall failed
no effective legacy probes attached
```

The failure also occurs when the collector is executed directly as root, so it
is not caused by the systemd pipeline, the Node.js forwarder, Source credentials,
or the service sandbox.

Inspection of the embedded `probes-legacy` object identified two Linux 4.19
compatibility violations:

1. `.BTF` contains a one-byte `c_void` enum rejected by the target kernel's BTF
   validator.
2. Compiler-generated `memcpy`, `memmove`, `memset`, `bcmp`, and `memcmp`
   subprograms contain backward branches. Linux 4.19 does not support bounded
   loops in eBPF programs.

## Selected Approach

Keep the current Rust userspace collector and its public behavior, but replace
the legacy probe build input with verifier-constrained BPF C. The legacy object
will be compiled without BTF and must not contain backward jumps. All fixed-size
operations will be explicitly bounded or unrolled so compiler output remains
acceptable to the Linux 4.19 verifier.

This is preferred over trying different Rust nightly versions because the Rust
compiler currently introduces loop-bearing memory builtins outside the source's
visible control flow. A C probe source gives direct control over sections,
helpers, map definitions, initialization, and unrolling while leaving the
well-tested Rust event enrichment/export path intact.

Kernel upgrade and service disablement remain operational alternatives, but
they do not satisfy the requirement to restore Observer telemetry on the current
host.

## Architecture

The collector remains one AArch64 ELF executable with the legacy BPF object
embedded at build time:

```text
kernel kprobe
  -> embedded BPF program
  -> perf-event array
  -> existing Rust legacy reader
  -> NDJSON stdout
  -> existing observer-forward.js
  -> AnySentry ingest API
```

The BPF C object preserves the program and map names consumed by
`a3s-observer-collector/src/legacy.rs`:

- Programs: `legacy_exec`, `legacy_exit`, `legacy_connect`, `legacy_setuid`,
  `legacy_ptrace`, `legacy_bind`, `legacy_openat`, `legacy_unlinkat`
- Perf maps: `EVENTS`, `EXIT_EVENTS`, `CONNECT_EVENTS`, `FILE_EVENTS`,
  `SEC_EVENTS`
- Per-CPU maps: `EXEC_SCRATCH`, `EXIT_SCRATCH`, `CONNECT_SCRATCH`,
  `FILE_SCRATCH`, `SEC_SCRATCH`, `DROPS`

The Rust userspace event structures and exported NDJSON schema do not change.
The collector continues to refuse healthy status unless at least one effective
probe attaches.

## Build Changes

The legacy build path will:

1. Compile a self-contained BPF C source for little-endian BPF.
2. Avoid kernel/libbpf development-header dependencies by defining only the
   required map ABI, context fields, and helper function signatures locally.
3. Disable BTF generation and remove `.BTF`/`.BTF.ext` defensively if the
   selected compiler emits them.
4. Fail the build when an executable BPF section contains a backward jump.
5. Fail the build when required programs or maps are missing.
6. Embed the verified object into the existing Rust collector.

The normal modern-kernel eBPF backend is not changed.

## Event Behavior

The hotfix retains the current legacy capture set:

- process execution and exit;
- outbound connect attempts;
- setuid, ptrace, and bind security signals;
- open/unlink file activity when `A3S_OBSERVER_FILES=1`.

Large events continue to use per-CPU scratch maps rather than the 512-byte BPF
stack. Fixed-size strings remain zero-terminated by clearing their first byte
before `bpf_probe_read_str`; event consumers already respect C-string
termination. The implementation will not clear whole large buffers with a
compiler-generated loop.

## Validation

Automated host-side validation must prove:

- source-level legacy contract checks pass;
- the BPF object contains all required programs and maps;
- the BPF object contains no `.BTF` or `.BTF.ext` sections;
- executable BPF sections contain no negative jump offsets;
- the AArch64 collector builds successfully against glibc 2.28;
- the collector ELF is AArch64, dynamically compatible with glibc 2.28, and
  safe on a 64 KiB page host;
- `a3s-observer-collector --version` identifies the legacy backend;
- the output binary and accompanying metadata have SHA-256 checksums.

Only the real target kernel can provide final verifier and kprobe-attachment
proof. Therefore the local artifact will be labelled a hotfix candidate until
the target-host checks below pass:

1. The collector remains running for at least 15 seconds when invoked directly.
2. Its stderr reports attached legacy probes rather than `BPF_PROG_LOAD` errors.
3. A generated process event appears on stdout.
4. The systemd Observer service becomes `active` after the binary is installed.
5. `/opt/anysentry/verify.sh` passes and the API event count increases.

## Delivery and Rollback

The delivery directory will contain:

- `a3s-observer-collector`
- `SHA256SUMS`
- build/source provenance metadata
- a target-host replacement guide

The replacement procedure will stop only `anysentry-observer.service`, copy the
existing collector to a timestamped backup, verify the candidate checksum,
install it atomically with mode `0755`, then run direct and service-level smoke
tests. Any failure restores the backup and restarts the previous service state.
ClickHouse and `anysentry.service` are not restarted.

## Non-Goals

- Upgrading or changing the target kernel.
- Modifying ClickHouse or the AnySentry API.
- Regenerating Observer Source credentials.
- Weakening collector health semantics to report healthy without an attached
  effective probe.
- Building the full AnySentry release archive before target validation.
